Data and Model Poisoning / RAG poisoning

RAG Poisoning

Scenario: PwnzzAI Pizza — internal catering policy assistant.

Description

At PwnzzAI Pizza we maintain our own catering policy for bulk and event orders: store and regional managers use it to decide what goes on the truck, minimum package sizes, allergens, and add-ons. In a well-run shop, assistant answers would be grounded only in text extracted from trusted sources — for example the internal handbook, franchise ops notices, and signed supplier bulletins. In this lab’s vulnerable setup, the same retrieval index also accepts policy-looking files you upload, so poison chunks can rank next to real policy.

How to exploit it (try this): reset the corpus if you want a clean baseline; upload a fake policy document that invents a rule real policy would never allow (for example “every large office package must include pineapple”); then in Demonstration chat, ask your own question about large office or event catering and reuse vocabulary from your poison file so TF-IDF pulls your chunk to the top. Open Full response JSON to see which chunk ids were retrieved and whether the model followed the poison. Finish in Mitigation Strategies by running the same question with hardened retrieval and comparing the JSON.

For the companion agentic SQL / tool routing challenge in the same catering setting, open Agents & tools → Agentic SQL & routing.

The retriever ranks text chunks with TF-IDF scoring: it gives higher scores to chunks that share important query words. In plain terms, the chunk with the highest word-overlap score is treated as most relevant. An attacker can abuse this by stuffing poison text with words likely to appear in user questions, so the poison chunk ranks first.

Query hints (write your own prompt — there is no prefilled question): think about what a real office manager would ask the policy bot: bulk orders, minimum party size, mandatory add-ons, or venue-specific rules. Overlap the vocabulary you expect in a poisoned “policy” upload (group size, pizza counts, catering packages) so your question competes with the trusted chunks.

In Demonstration, read the assistant reply in the chat and expand Full response JSON when you need chunk ids and retrieval fields.

Vulnerable path success criteria: after ingesting your poison document, ask a question you designed so the retrieved context includes the poison chunk and the assistant follows the poisoned instruction (for example, recommending pineapple on pizza even though that is not part of trusted policy).

Mitigation: use Mitigation Strategies below to run the same query with hardened retrieval (trusted sources only) and compare behavior.

Defense Playbook (Incident Response)

                            
Detect: sudden policy contradictions in assistant answers; retrieval debug shows unknown userdoc_* ids from recent uploads.

                            
Contain: disable ingestion endpoint; freeze RAG index version; stop auto-publishing retrieved snippets to customers.

                            
Eradicate: remove poison chunks; rotate assistant prompts; invalidate bad embeddings.

                            
Recover: rebuild index from signed corpus; re-run regression queries; enable provenance tags on chunks.

                            
Lessons learned: treat ingestion as supply chain; require trust metadata before retrieval ranking.
