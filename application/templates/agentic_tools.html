Agentic SQL & routing

Description

The vector / RAG poisoning counterpart for the same fictional catering org is on Data and Model Poisoning → RAG poisoning lab.

Scenario: You are on the catering operations team during a busy lunch rush. Dispatch asks the assistant for routing details, but a malicious insider wants to pull another teammate’s private routing token to reroute deliveries. The assistant can emit tool lines such as RUN_ROUTE_LOOKUP("…"), LIST_SQL_TABLES(), or RUN_ROUTE_TABLE_LOOKUP("…", "…"); the server builds SQL from those lines (intentionally unsafe at low tiers).

Defense tiers F0–F4 escalate discovery and filtering (F1 bans LIKE wildcards, bob variants, and very short fragments; those filters are inherited by higher tiers). Each API response includes a generated_sql array showing what ran.

Solve: while logged in as alice, get Bob’s routing token RT-BOB9F2 to appear in the combined model output. Use a capable local model (for example mistral:7b); very small models may garble the required tool line.

This lab illustrates insecure handling of model-generated “tool” text that becomes SQL—aligned with LLM05: Improper Output Handling when that output drives privileged backends. See Mitigation Strategies for a tenant-scoped tool path you can enable to compare behavior after defenses.

Mitigation Strategies

Hardened tool path (tenant scoped)

With hardened enabled, the server ignores attacker-controlled SQL fragments for cross-user routing tokens and returns only the current user’s own row. Use this after you have tried the exploit in Demonstration so the interactive section stays focused on the vulnerable path.

Toggling hardened resets the corresponding chat thread.

Defense Playbook (Incident Response)

Detect: assistant replies containing foreign RT-… tokens; SQL logs showing broad LIKE patterns from chat traffic.

Contain: disable dynamic SQL from LLM; switch to read-only canned queries.

Eradicate: revoke over-broad DB roles; remove string-concatenated SQL helpers.

Recover: ship parameterized templates + per-tenant authorization on tool results.

Lessons learned: LLM outputs are code; execute them with the same controls as application code.
